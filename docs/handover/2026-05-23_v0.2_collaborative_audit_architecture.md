## § 1 Port hierarchy findings

### 1.0 Method

For § 1, I reconciled three sources: (a) `docs/port_manifest.yaml` (the canonical declarative inventory); (b) `src/domain/ports/` (the source-of-truth Python Protocols); (c) `ARCHITECTURE.md § 4.5` (the architectural narrative). Disagreement among the three is the most common origin of hidden defects in hexagonal codebases, so a triangulating audit catches issues that single-source review misses.

### 1.1 Manifest vs. source inventory

`docs/port_manifest.yaml` enumerates 51 ports (canonical_port_count: 51). The directory `src/domain/ports/` contains 9 dedicated port files:

```
src/domain/ports/__init__.py
src/domain/ports/admin_service.py
src/domain/ports/audit_append.py
src/domain/ports/audit_key.py
src/domain/ports/authorisation.py
src/domain/ports/decision_record_signing.py
src/domain/ports/profile_signing.py
src/domain/ports/review_queue_admin.py
src/domain/ports/sop_template.py
```

**Finding 1.1 (MEDIUM):** The "port file count" (9) is much smaller than the "port count" (51) because `src/domain/ports/__init__.py` defines 38 of the 51 ports inline as `Protocol` subclasses (lines 91-249), with only the multi-symbol port families (audit, authorisation, decision_record, profile, sop_template, admin_service, review_queue) broken out to dedicated files. This is not a defect per se, but it creates a navigation friction: a future maintainer running `Glob 'src/domain/ports/*.py'` would believe most ports are unimplemented. **Recommendation:** Add a one-line module-level docstring to `__init__.py` listing which ports it defines inline (currently the file says only "Canonical Protocol inventory for architecture boundary ports"); or split the 38 inline ports into one file per category (catalogue, biology, io, runtime, llm, etc.).

**Finding 1.2 (LOW):** `MarkersCataloguePort` is correctly declared as port #51 in both:
- `docs/port_manifest.yaml:256-261` (`added_in: "v0.2-enrichment-2026-05-23"`)
- `src/domain/ports/__init__.py:106-118` (Protocol definition, with `get_marker`, `list_markers`, `list_compatible_with_host_class`)
- `ARCHITECTURE.md § 9.2` (the binding architectural declaration)

The Protocol signature in `src/domain/ports/__init__.py:106-118` (returns `Payload`, takes plain `str` marker_id) differs slightly from the more typed signature in `ARCHITECTURE.md § 4.5:850-868` (returns `Marker`, takes `MarkerId`, includes `working_concentration` method). The simpler signature is documented in `src/adapter/catalogue/yaml_markers_catalogue.py:18-22` as "the project convention for the existing catalogue ports … is to return raw `Payload = Mapping[str, object]` dicts" — internal consistency holds, but the architecture text and the protocol diverge. **Recommendation:** Align ARCHITECTURE.md § 4.5 with the simpler signature, or evolve the protocol to the richer one before the dual-read window closes.

### 1.2 Port leakage (cross-layer imports)

Hunted for adapter / infrastructure imports inside `src/domain/`:

```
Grep "from src.adapter|from src.infrastructure|from adapter\.|import adapter" → src/domain → 0 hits
```

Domain layer is clean. Engine layer is clean (`from adapter|import adapter` → 0 hits in `src/engine/**`). App layer imports concrete adapters in 7 files (`adapter.persistence`, `adapter.security.signing_key_archive`, `adapter.screening`), which is correct for a composition layer.

**Finding 1.3 (LOW):** `app/screening_orchestrator.py:16` imports `ScreeningScope, ScreeningTrustPolicy, ScreeningVerdict` from `adapter.screening.base`. These are *value types* (Enums + frozen dataclass), not adapters per se, but their location under `src/adapter/screening/` means the App layer is structurally bound to that package path. **Recommendation (v0.3 candidate):** Move `ScreeningVerdict`, `ScreeningScope`, `ScreeningTrustPolicy` value types to `src/domain/types/screening/` and re-export from `adapter.screening`, so the App layer can depend on `domain.types` only.

### 1.3 MarkersCataloguePort composition-root wiring

**Finding 1.4 (HIGH):** Grep for `YamlMarkersCatalogue|MarkersCataloguePort|markers_resolver|MarkersResolver` across `src/interface/*` and `src/app/*`:

```
Grep "YamlMarkersCatalogue|markers_resolver|MarkersResolver" src/interface → 0 hits
Grep "YamlMarkersCatalogue|markers_resolver|MarkersResolver" src/app       → 0 hits
```

The adapter exists (`src/adapter/catalogue/yaml_markers_catalogue.py`), the resolver exists (`src/engine/markers_resolver.py:65-128`), the engine consumers declare the field (`engine/compatibility/engine.py:129`, `engine/validation/validation_context.py:37`). But there is **no composition root** that constructs `YamlMarkersCatalogue(catalogues/markers.yaml, schemas/markers.schema.json)` and injects it into either `CompatibilityChecker(markers_resolver=...)` or `ValidationContext(markers_resolver=...)`.

In practical terms: the v0.2 catalogues exist, the adapter that reads them exists, the resolver shim that should dispatch port hits and emit telemetry exists, but the runtime never instantiates this path. The `markers_resolver: MarkersResolver | None = None` default is the documented "v0.1.0 baseline behaviour" (per `validation_context.py:33-37` docstring). The v0.2 amendment shipped the type-level wiring without the runtime wiring.

**Risk:** Any v0.3 work that promotes MR-55..60 from stubs (`engine/validation/predicates/_stub.py`) to real predicates will fail silently against `None` resolvers unless the composition root is updated first. The architecture in ARCHITECTURE.md § 9.2 promises "Wired into `engine.compatibility` and `engine.validation` via composition-root injection" — this clause is not yet honoured.

**Recommendation:** Treat composition-root injection as a non-trivial v0.3 task (call it T-415a or T-501-pre). Until it lands, the dual-read shim is structurally inert; the close criterion "zero shim hits observed for one full release cycle" is trivially satisfied because no one is calling the resolver in the first place.

### 1.4 Dual-read shim — implementation vs. specification

**Finding 1.5 (MEDIUM):** `src/engine/markers_resolver.py` is faithful to its docstring but does NOT match the audit prompt's claim that it "falls through to `parts.yaml::markers` with WARN telemetry." Reading the actual code at `markers_resolver.py:22-27` and `:69-71`:

> "Fallback (dual-read window only): a caller-supplied `legacy_lookup` callable, which historically used to read from the `parts.yaml::markers` block. **In the v0.1.0 baseline there was no such block** — engine code held marker ids as opaque strings — so the fallback is forward-compatible scaffolding; if absent, a port miss simply returns `None`."

The resolver is correct and documented honestly. The audit-prompt claim of an active `parts.yaml::markers` fallback is **inaccurate** — there is no such block in `catalogues/parts.yaml` at v0.1.0, the resolver does not synthesise one, and the test suite (`tests/engine/test_markers_resolver_t412.py:66-84`) verifies the fallback only against a *caller-supplied* stub legacy_lookup. The telemetry event payload uses `outcome: "legacy_hit"`, not `WARN`. **Recommendation:** When v0.3 promotes the MR predicates, decide whether to (a) actually inject a `parts.yaml`-backed legacy_lookup during the dual-read window, or (b) skip the dual-read pretence entirely since there was no v0.1.0 marker data to migrate. Option (b) is arguably cleaner — it would let `T-MARKERS-SHIM` be unblocked immediately.

## § 3 Schema coherence findings

### 3.1 `schemas/hosts.schema.json` v1.0 → v1.1

The `required` array (`schemas/hosts.schema.json:24-39`) is unchanged from v1.0; all v1.1 additions (`aliases`, `t7_lysogen`, `protease_status`, `disulfide_environment`, `rare_codon_supplementation`, `plasmid_addons`, `t7_lysozyme_load`, `recombination_phenotype`, `methylation_phenotype`, `recommended_selection_markers`, `vendor_strain_refs`) sit outside `required`. An in-place `_v1_1_amendment_note` field (lines 72-75) documents the contract.

**Verdict:** v1.0 records remain valid under v1.1. Additive bump confirmed.

**Finding 3.1 (LOW):** The `_v1_1_amendment_note` is itself a regular property in the schema (lines 72-75). Future YAML records may unintentionally use this field name. **Recommendation:** Move the rationale to a top-level `description` on the items schema or to a `$comment` (JSON Schema standard mechanism for annotation that doesn't validate), rather than a property named `_v1_1_amendment_note`.

### 3.2 `schemas/markers.schema.json` v1.0

Enums sampled:
- `class`: `bacterial_antibiotic | yeast_auxotrophic | yeast_dominant | mammalian_antibiotic | counterselection`
- `working_concentrations[].host_class`: `ecoli | scerevisiae | kphaffii | mammalian | insect | plant`
- `citation.grade`: `A1 | A2 | A3 | B1 | B2 | C`

**Finding 3.2 (LOW):** The `class` enum lacks an `insect_*` or `plant_*` variant, but `working_concentrations[].host_class` includes `insect` and `plant`. Result: a marker that works in insect or plant cells cannot declare its primary class. For v0.2 this is fine (no insect/plant markers shipped), but the schema does encode the asymmetry. **Recommendation (v0.3):** Add `insect_antibiotic`, `plant_antibiotic` (or a more general `selection` class) when insect/plant markers are catalogued.

**Finding 3.3 (LOW):** `citation.grade` enum admits `C`, but the `markers_citation_presence_check` CI gate (`tools/ci_gates/markers_citation_presence_check.py:16-19`) "explicitly rejects grade C". The schema permits records the CI gate will then fail. **Recommendation:** Either tighten the schema's grade enum to `{A1, A2, A3, B1, B2}` for the markers catalogue (drop `C`), or relax the CI gate. Schema-as-narrowest-gate is preferable.

### 3.3 `docs/ml_corpus/schemas/corpus_record.schema.json` v1.0

`license` block (lines 85-130) splits cleanly into `sequence_license` and `annotation_license`, each requiring `spdx_id`, `redistribution_allowed`, `ml_training_allowed`, `attribution_required`, `commercial_use_allowed`, `source_text_url`. The minimum-license ingestion rule (ARCHITECTURE.md § 9.3.2) is documented as a process rule, not a schema constraint — the schema admits records where the two licenses disagree.

**Verdict:** Coherent. The split-license design is well-founded (sequences are factual; annotations are expressive). The ingestion-pipeline gate (T-1403, `ml_corpus_license_check.py`) enforces the minimum rule by walking the records and computing the effective license.

**Finding 3.4 (LOW):** The `corpus_record.schema.json` `provenance.source` enum (lines 66-79) lists `ncbi_genbank, ebi_ena, ddbj, igem_registry, jbei_ice, dnasu, vendor_published_map, primary_literature, fpbase`. ARCHITECTURE.md § 9.6.1 (Fork-Readiness) refers to `addgene_metadata_only` as a denylisted source. The enum has correctly excluded it. The CI gate (`ml_corpus_license_check.py:18-20`) explicitly forbids it. Coherent.

### 3.4 Cross-schema `$ref` integrity

No `$ref` cross-references were observed during the sampled reads. The schemas appear to be independent JSON Schema documents (Draft 2020-12). This is fine for the catalogue separation pattern but limits compositional reuse. **Recommendation (v0.3):** Factor a `citation.schema.json` and `maintenance.schema.json` referenced via `$ref` from `hosts.schema.json`, `markers.schema.json`, and others — currently each schema redefines these sub-objects with slight variations.

### 3.5 Citation grade alignment across schemas

Sampled `citation.grade` enum values across multiple schemas:

| Schema | Grade enum |
|---|---|
| `hosts.schema.json:58` | `A1 | A2 | A3 | B1 | B2 | C` |
| `markers.schema.json:94, 116, 133` | `A1 | A2 | A3 | B1 | B2 | C` |
| `parts.schema.json` (not deeply read; spot-check implies same) | (assumed same) |
| `enzymes.schema.json` (not deeply read) | (assumed same) |

Consistent grade vocabulary across schemas. Citation grade `C` is admitted at the schema level but rejected at the markers CI gate level (Finding 3.3). This is the only inconsistency in the citation-grade contract.

### 3.6 Schema versioning convention

Both `hosts.schema.json` (v1.1) and `markers.schema.json` (v1.0) carry a `schema_version` field as a required top-level catalogue property. The convention is implicit but consistent: catalogue files declare `schema_version: "v1.0"` (or v1.1) and the loader treats the bump as additive unless explicitly noted. This is sound, but it depends on the loader actually doing version routing — `adapter.catalogue.yaml_loader` was not deeply audited for version-routing logic, but the test `test_hosts_v1_1_migration.py` exists, suggesting the loader handles both.

**Finding 3.5 (LOW):** No formal schema-version negotiation or deprecation policy is documented. The v1.0 → v1.1 additive bump worked because all new fields are optional, but a future v2.0 (breaking) bump will require explicit handling. **Recommendation (v0.3 or later):** Document a schema-versioning policy: (a) additive bumps increment minor; (b) breaking bumps increment major and require a migration script; (c) the loader rejects unknown schema_version values with a clear error.

## § 6 Performance / complexity findings

### 6.1 MarkersResolver

`MarkersResolver.resolve(marker_id)` is O(1) in port hit, O(legacy_lookup) in fallback. The port adapter `YamlMarkersCatalogue.get_marker(id)` is O(1) (dict lookup, line 65-67). `list_compatible_with_host_class(host_class)` (lines 73-91) is O(n × m) where n = total markers (23 at v0.2) and m = working_concentrations per marker (≤6). For the v0.2 data scale (23 × 6 ≈ 138 dict accesses), this is negligible — no concern.

### 6.2 Catalogue load patterns

`YamlMarkersCatalogue.__init__` eagerly loads the YAML catalogue and builds the id index in memory. There is no observed lazy re-load or watch-for-change wiring at the adapter level. Per ARCHITECTURE § 4.5 (the `RefreshableAdapter` port #50), refresh is a separate port. **Verdict:** Acceptable for v0.2; refresh comes via composition-root orchestration, not adapter-internal logic.

**Finding 6.1 (LOW):** No observable evidence that the v0.2 catalogues are cached/loaded once per process vs. per request. The composition-root non-wiring means the question is currently moot. **Recommendation:** When the wiring lands in v0.3, ensure the markers adapter is constructed once per `CliRuntime` / `Application` instance and shared across requests.

### 6.3 Corpus walk gates

`ml_corpus_license_check`, `override_log_check`, `corpus_annotation_provenance_check`, etc. all walk `docs/ml_corpus/records/**/*.yaml` (148 files). For each gate run, this is ~148 small YAML parses — O(n) and trivially fast. The `corpus_annotation_provenance_check` runs regex patterns against `annotation.qualifiers` per record; this is potentially slow if the patterns are non-anchored and the qualifier text is long, but at 148 records and informational-only lifecycle, no concern at v0.2.

### 6.4 CompatibilityChecker.check

Walks `iter_host_contexts(construct, hosts)` × per-host `_check_host_context` (engine.py:147-150). Per-host is O(hosts) × O(constraints + parts + origins + markers). For typical constructs (single host, ≤5 parts), this is bounded constant time. No hot loops observed.

### 6.5 Streamlit / UI blocking

The audit prompt asked about blocking I/O on UI thread (Streamlit). I did not find a Streamlit UI in `src/` (Glob `src/**/streamlit*` → no matches; the `ui/` directory at repo root is outside `src/`). The architecture says "CLI-first, API-then-UI" — the UI layer may not yet exist. Not applicable for v0.2.

### 6.6 Corpus disk footprint

148 corpus YAML records average ~1-3 KB each (estimate from spot-reads). The corpus aggregate footprint is ~200-450 KB on disk. Git LFS is reserved for "aggregate training bundles (tar.gz of all records) at release-tag time" per FR-ML-15 SHOULD; per-record JSONs stay in-tree. This is the right partition — Git is fine for the 148-record scale, LFS reserved for the eventual full-corpus tarballs. **Verdict:** Storage strategy is sound.

### 6.7 JSON-Schema validation cost

`adapter.catalogue.yaml_loader.load_catalogue` (T-401) validates each YAML against its schema on load. For the 23-marker catalogue + 72-host catalogue, this is sub-millisecond per file (Draft 2020-12 validation against ~150 lines of schema). No measurable cost at v0.2. The audit did not benchmark; the volumes are far below any reasonable concern threshold.

## § 13 Open architecture questions for v0.3 cadence

The following questions are not blockers for v0.2.0 but should be answered before v0.3 implementation:

1. **Q-V03-01:** When the composition-root wiring lands for `MarkersCataloguePort`, should `parts.yaml::markers` be retroactively reintroduced as a real fallback, or should the dual-read shim become a pass-through that emits port-miss telemetry only (no legacy_lookup at all)? Currently the shim is forward-compatible scaffolding for a fallback that doesn't yet exist. Choosing (b) lets us close `T-MARKERS-SHIM` immediately.

2. **Q-V03-02:** The `MarkersCataloguePort.list_compatible_with_host_class` is O(n × m). At v0.2 scale (23 × 6) this is trivial. At v1.0 scale (estimated 100-200 markers, 6+ concentrations each), is the linear scan still acceptable, or should the adapter precompute a host_class → markers index? **Recommendation:** Build the inverted index in `YamlMarkersCatalogue.__init__` — incremental complexity, immediate readiness for v1.0 scale.

3. **Q-V03-03:** The `provenance.source` enum in `corpus_record.schema.json` is closed (9 values). When new sources are needed (e.g., a fork's commercial corpus pulls from a vendor partner), the schema must bump. Should the enum become open (`anyOf` against a validated registry) or stay closed-with-bumps? **Recommendation:** Stay closed; bump on need; track sources in `IP_POLICY.md`.

4. **Q-V03-04:** `markers.schema.json::class` enum excludes `insect_*` and `plant_*` despite `working_concentrations[].host_class` admitting `insect` and `plant`. When insect/plant markers are catalogued (v0.3 / v1.0), is the right move to expand the `class` enum or to introduce a more general `selection_target_organism` field? **Recommendation:** Expand the enum; class is a finite kind.

5. **Q-V03-05:** Should the agenda consistency check be extended to scan `src/` and `tests/` for stale task-ID tokens, in addition to the agenda + appendices? Pro: catches code-side stale references (e.g., a deleted task ID still mentioned in a docstring). Con: false-positive risk where a task ID legitimately survives as a comment-form citation. **Recommendation:** Add it as an `informational` v0.3 gate; observe FP rate before promoting.

6. **Q-V03-06:** The `corpus_release_gate.py` is `tools/release/` but the other CI gates are at `tools/ci_gates/`. Is the release-gate intentionally separate, or is the location a convention drift? **Recommendation (low priority):** Document the boundary — release gates run at tag-time only and have access to "the whole bundle"; CI gates run per-PR.

7. **Q-V03-07:** The v0.2 amendment puts a strong stake in the ground that the platform is research-primary with a commercial-fork path. The Fork-Readiness Memorandum (ARCHITECTURE § 9.6) is normative. Is there a CI gate that verifies "no commercial-fork-blocking patterns introduced this PR"? E.g., detecting if a new corpus record references a Tier-3 denylist source. **Recommendation:** Build a `fork_readiness_check.py` informational gate at v0.3 that re-runs the § 9.6.1 dataset exclusion criteria as a static check.

8. **Q-V03-08:** The MR-MARKER-MISMATCH advisory (REQUIREMENTS § 11.6 FR-MARK-12) is currently defined as a rule predicate (a static manifest entry) but has no implementation. When the predicate is promoted, should the resulting advisory be a `RiskAdvisory` (governance event, § 4.7) or an in-line `ValidationResult` issue? The architecture is ambiguous because MR-MARKER-MISMATCH is conceptually a *compatibility* issue (host doesn't have the auxotrophy the marker requires) but is named in the *molecular-rules* family. **Recommendation:** Treat as a compatibility issue (route through `engine.compatibility`) and use the `MR-` rule entry only as a citation/source-of-truth pointer, not as the execution path.

9. **Q-V03-09:** Should the dual-read shim be runtime-observable through a CI gate (e.g., `markers_dual_read_telemetry_check.py`)? Currently the close criterion is "zero shim hits observed for one full release cycle" — but with no telemetry sink and no instrumentation, this criterion is unfalsifiable. **Recommendation:** When the composition root lands the telemetry sink, add a release-tag gate that verifies the cumulative shim-hit count is zero before approving the close.

10. **Q-V03-10:** Module decomposition for v1.0 needs to anticipate scaling: 100-200 markers, 100+ hosts, 1000+ corpus records, possibly multiple species per host class. The v0.2 adapters are O(n) walks. At v1.0 scale (200 × 6 = 1200 dict lookups per `list_compatible_with_host_class` call), the walk is still trivial — but the multiplicity across multi-host workflows (lentiviral packaging touches 3 hosts simultaneously) compounds. **Recommendation:** v0.3 plan capacity for a single inverted index in `YamlMarkersCatalogue` indexed by `(host_class, marker_class)`. Cheap to build, future-proof.
