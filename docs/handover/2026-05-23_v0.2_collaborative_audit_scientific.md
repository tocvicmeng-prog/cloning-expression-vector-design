## § 0. Executive verdict

**Verdict: PASS_WITH_CONCERNS.**

The v0.2 toolkit demonstrates substantial molecular-biology depth and produces wet-lab-compatible blueprints in industry-standard formats (GenBank via Biopython, SBOL3 via pySBOL3). The 60-rule MR catalogue and 35-host catalogue cover the core biology that a vector-design engine needs: central-dogma constraints, host/marker compatibility, codon usage, translation initiation, polyadenylation, splicing, CpG silencing, signal-peptide routing, methylation-aware enzyme planning, disulfide-aware host selection, protease awareness, rare-codon supplementation, auxotrophic-marker matching, and Y2H reporter avoidance. The compatibility engine correctly consults `host.compatible_origins`, `host.compatible_markers`, and `part.host_compatibility` set membership, and the assembly strategies (Gibson, NEBuilder HiFi, In-Fusion, Golden Gate, MoClo, YTK, GoldenBraid, Loop, JUMP, MIDAS, restriction-ligation, IVA, LIC, SLIC, USER, yeast TAR, Gateway) ship with bench-realistic enzyme/buffer/cycling parameter slots and Pryor-2018/Potapov-2020 overhang-fidelity scoring. Sequences are written with topology preserved, features round-trip cleanly, and SHA256 byte-equality verifications against the canonical source were performed on 78/148 corpus records with zero mismatches.

However, three classes of concern keep the audit from full PASS:

1. **The rule catalogue's `implementation_status` is uniformly "stub" across all 128 BR/MR/MS/SR/WR entries**, even though ~70 predicates are now actually wired through `DEFAULT_VALIDATION_PREDICATES` in `src/app/validation_orchestrator.py`. This stale metadata makes it impossible for a reader to tell which biological rules will actually fire at runtime.

2. **A semantic mismatch between predicate name `mr_22` and MR.yaml rule MR-22.** The implemented predicate `mr_22_forbidden_internal_sites` (in `src/engine/validation/predicates/internal_sites.py`) checks for forbidden restriction sites; MR.yaml rule MR-22 declares the semantics as "If wild-type WPRE detected → recommend WPRE3 (mut6) variant." Same predicate name, different biology. If both registries are dict-merged, the wrong-semantics predicate will run for MR-22.

3. **Specific corpus defects that would damage a wet-lab user.** The polyA record `sv40.yaml` contains the BGH polyA sequence (not SV40 — bytes verified essentially identical to `bgh.yaml`); the FP record `EYFP_AB860250.yaml` contains a Mycoplasma-codon-optimised CDS with an in-frame `TGA` at codon 58 (interpreted as Trp in NCBI table 4 but as a stop in table 1/11) but is labelled `host_class: multi` and `transl_table: '11'`; multiple PMIDs in citations point to unrelated papers (`6328430` is RK2 plasmid biology, not BGH polyA; `21909101` is feline antiviral transgenesis, not P2A; `25244104` returns 404; `6296021` is cefazolin surgical prophylaxis, not neoR); the yeast host genotype field has been reduced to placeholder strings like "S288C-derived haploid auxotrophic background; verify source genotype" which prevents the MR-59 host_genotype_requirement match from succeeding because the canonical tokens `ura3-Δ`, `leu2-Δ`, `his3-Δ`, `trp1-Δ`, `met15-Δ` are not present in the host record.

None of these issues, individually, threatens v0.2 release viability; collectively they require a targeted v0.2.1 corpus + rule-metadata fix before the corpus can be safely used as ML training data or before MR-22 / MR-59 can be advertised as "active."

## § 1. Wet-lab executability findings

### § 1.1 Output formats import cleanly into bench tooling — HIGH confidence pass

`src/adapter/io/genbank.py` wraps `Bio.SeqIO.write(...)` for GenBank, and `src/adapter/io/sbol3.py` wraps `pysbol3` for SBOL3 NTriples. Roundtrips are tested. Topology is preserved via `record.annotations["topology"]`, features round-trip from Biopython `SeqFeature` to the project's `FeatureV14` model and back (`src/adapter/io/_biopython.py`). The SBOL3 adapter uses the canonical SO ontology mappings (`SO_CDS`, `SO_GENE`, `SO_PROMOTER`, `SO_RBS`, `SO_TERMINATOR`), so feature roles will be understood by Benchling, SnapGene, and ApE on import.

Biopython's GenBank writer enforces `LOCUS` line discipline (16-char ID, length, molecule type, topology, division), the `FEATURES` block with `/qualifier="..."` formatting, and the `ORIGIN` block with the canonical 60-char/line breakdown plus 10-char column grouping. SnapGene and Benchling read this format losslessly. ApE reads GenBank but loses some SBOL-only metadata (e.g., SO ontology URIs) — acceptable because SnapGene and Benchling are the primary consumers.

The `WriteResult` dataclass carries a `semantic_fingerprint` (SHA256 of the canonical sequence + ordered feature list) which allows users to verify that a roundtrip GenBank → SBOL3 → GenBank preserved semantic content even if the line-level formatting changed. This is the kind of defensive bookkeeping a wet-lab user needs to detect "vendor stripped my annotation" failures.

**No HIGH-severity executability defects observed in I/O.**

### § 1.2 Assembly strategies are bench-realistic — PASS

`src/engine/assembly/gibson.py`, `golden_gate.py`, `restriction_ligation.py` carry the wet-lab parameters one needs to actually run the protocols:

- **Gibson-like (Gibson / NEBuilder HiFi / In-Fusion):** `default_overlap_length` 20 nt (Gibson, NEB) or 15 nt (In-Fusion), `verification_checkpoints=("junction_sanger",)`, `expected_failure_modes=("insufficient_overlap", "template_carryover")`. These are the failure modes that actually occur at the bench.
- **Golden Gate (BsaI-HFv2, BsmBI-v2, BbsI-HF variants):** `cycling_profile = "30 cycles 37C/16C, final heat inactivation"` (canonical Engler/Marillonnet kinetics); Potapov-2018/Pryor-2020 overhang-fidelity scoring via `OverhangSetOptimiser` with `minimum_fidelity=0.90`, `minimum_per_overhang_fidelity=0.99`, `max_pair_crosstalk=0.02`. These thresholds match published Golden Gate best practice.
- **Restriction-ligation:** default `T4 DNA ligase, compatible cohesive ends`; dephosphorylation flag exposed; `verification_checkpoints=("diagnostic_digest", "junction_sanger")`; `expected_failure_modes=("vector_religation", "wrong_insert_orientation")`. Both failure modes are real, well-known, and addressable (CIP/SAP treatment, directional cloning).

### § 1.3 Reading-frame predicate — PASS (with one nit)

`src/engine/validation/predicates/frame.py::_valid_coding_sequence` correctly enforces: length ≥ 6, length % 3 == 0, first 3 nt in `{ATG, GTG, TTG}`, last 3 nt in `{TAA, TAG, TGA}`, no internal stops. MR-11 tandem-stop predicate enforces `sequence.endswith("TAATAA")`. These match the MR-10/MR-11 declared semantics in `catalogues/rules/MR.yaml` lines 396–435.

**Minor nit (MEDIUM):** the predicate accepts `GTG` and `TTG` starts unconditionally, but MR-10's description says "ATG (or GTG/TTG in E. coli when explicitly declared)". The predicate has no host-class gating, so a mammalian construct using a non-standard `GTG` initiator would silently pass MR-10 even though, in a Kozak-scanning mammalian system, `GTG` initiation is exceptional. Recommendation: gate non-ATG starts on host chassis (or on an explicit `non_standard_start_codon: true` payload flag).

### § 1.4 Restriction-site scanning is biologically correct — PASS

`src/engine/sequence_analysis/find_sites.py` implements topology-aware IUPAC restriction-site finding with full ambiguity codes (R, Y, S, W, K, M, B, D, H, V, N). It handles `wraps_origin` for circular plasmids (critical because many cut sites in canonical backbones lie across the origin coordinate; missing the wrap-around site is a common bug in naïve restriction-mappers). The `RestrictionEnzyme` dataclass validates `cut_index` against site length and rejects unsupported symbols. Topology defaults follow the parent `SequenceRecord`.

The IUPAC table is mathematically complete for restriction enzyme recognition: `IUPAC_BASES` covers all 11 ambiguity codes plus the 4 pure bases. Each ambiguity code maps to its biologically correct base subset. This matches REBASE convention and IUBMB nomenclature. A wet-lab user supplying `GGCCNNNNNGGCC` (BglI) or `CCWGG` (EcoRII) gets correct matches.

The `find_sites_on_strand` implementation correctly handles palindromic enzymes (single strand scan + flag) and non-palindromic enzymes (scan both strands and report strand orientation). For Type IIS enzymes (BsaI `GGTCTC(1/5)`, BsmBI `CGTCTC(1/5)`, BbsI `GAAGAC(2/6)`), the `cut_index` correctly displaces the cut position outside the recognition sequence — which is the defining feature of Type IIS biology and the basis of Golden Gate scarless assembly.

### § 1.5 EYFP_AB860250 is a Mycoplasma-codon CDS with an in-frame TGA — HIGH-severity

`docs/ml_corpus/records/elements/fluorescent_proteins/EYFP_AB860250.yaml` stores a 720-bp EYFP CDS with `host_class: multi`, `intended_use_category: [fluorescent_reporter, yellow_fp, monomeric]`, and `transl_table: '11'`. In-frame codon-by-codon scan shows an internal `TGA` at position 172 (codon 58) and the canonical terminal `TAA` at position 718.

NCBI record `AB860250` (cross-verified via `eutils.ncbi.nlm.nih.gov/entrez/eutils/efetch.fcgi?db=nuccore&id=AB860250&rettype=gb&retmode=text`) discloses this as a **Ureaplasma parvum**-optimised EYFP gene, with a note about "Tryptophan at positions 172-174." Ureaplasma is in NCBI translation table 4 (Mycoplasma/Spiroplasma) where `TGA` codes for Trp, not stop. In NCBI table 11 (bacterial standard) or table 1 (vertebrate cytoplasm), the `TGA` is a stop codon and the protein will be truncated at codon 58 — yielding a 57-aa peptide, not the 239-aa EYFP fluorophore. The fluorophore chromophore-forming Tyr66-Gly67-Tyr65 triad would not even be reached.

**Impact:** a user pulling this EYFP record into a mammalian or E. coli expression construct will get no fluorescence. The corpus does not signal the Mycoplasma codon table issue. The `transl_table` field is wrong.

**Recommendation (v0.2.1):** either (i) replace this record with a standard-table EYFP CDS (e.g., GenBank `KX808521` or `AB915953` if standard-codon-optimised); (ii) reclassify `host_class: mycoplasma` and add an explicit `transl_table: '4'` with a `notes:` flag; or (iii) move this record to `exclusions.yaml` with reason `sequence_legitimacy_in_doubt` until a wet-lab user reviews.

### § 1.6 BBa_E1010 (mRFP1) contains the BioBrick suffix scar — MEDIUM

`docs/ml_corpus/records/elements/fluorescent_proteins/BBa_E1010_igem.yaml` is 706 bp but `706 mod 3 == 1`. The CDS proper ends at the tandem `TAATAA` around position 676–681; bases 682–706 are the iGEM BioBrick BBb suffix scar `CGCTGATAGTGCTAGTGTAGATCGC`. The record carries `annotation: []`, so there is no indication to the user that the CDS only spans 1..681 and that 682..706 is part-construction scar.

**Impact:** any tool that tries to translate the full 706-bp record will read a partial trailing codon and produce a gibberish translation suffix. Any tool that infers `length_bp % 3 == 0` as a CDS sanity check will reject this record. A wet-lab user expecting "mRFP1 CDS for re-cloning" will need to trim the BioBrick suffix.

**Recommendation:** add explicit annotations (`CDS 1..681`, `misc_feature 682..706 /label="BioBrick suffix scar"`) to clarify what's CDS and what's part-construction overhead.

### § 1.7 mScarlet (KY021423), mTagBFP2 (MH013372), mRuby3 (KX987299) are partial CDSes — LOW

These three FP records faithfully mirror their NCBI deposits, which are partial CDSes (CDS `1..>696`, `1..>699`, `1..>711` respectively). The bases end without a stop codon. A user expecting "complete CDS" will get a protein lacking the natural C-terminus and the terminal stop. This is **not a corpus bug** — the canonical INSDC record is partial — but the corpus also does not surface "partial CDS" in any human-readable field.

**Recommendation:** add a `cds_completeness: partial | complete` field to the corpus schema and populate it from the GenBank `CDS  1..>N` partial-feature marker.

### § 1.8 pTrc99A topology stored as linear — LOW

`pTrc99A_M22744.yaml` carries `topology: linear`, but pTrc99A is a circular cloning plasmid (Amann 1988). The stored topology mirrors the NCBI record M22744 (which itself omits the topology flag for many vintage cloning-vector deposits). Downstream tools that assume the topology field is authoritative may treat the plasmid as linear for restriction-site / origin-of-replication analyses, missing sites that wrap the origin coordinate. Same issue may apply to M13mp18 X02513.

**Recommendation:** add a corpus-level override field `topology_authoritative: circular | linear | as_deposited` so curator domain knowledge can correct NCBI's `linear` defaults where the biological molecule is canonical circular.

### § 1.9 BBa_E0040 (GFPmut3b) tandem stop — PASS

720 bp, ends `AAA TAA TAA` (Lys, stop, stop). Tandem stop pattern is exactly what MR-11 wants. Frame is clean.

### § 2.3 MR-22 predicate ↔ rule semantic mismatch — HARD/HIGH defect

- **Rule MR-22 (MR.yaml lines 822–858):** description = "If wild-type WPRE detected → recommend WPRE3 (mut6) variant"; severity = SOFT; citation = Zanta-Boussif 2009 PMC; Schambach 2006.
- **Implemented `mr_22` predicate (internal_sites.py lines 17–36):** scans the construct's `sequence` payload for `forbidden_restriction_sites[].recognition_site` matches and returns `rule.severity` if any are found.

These are biologically unrelated checks. WPRE detection is a sequence-identity check against the WPRE wild-type vs WPRE3 (mut6, Zanta-Boussif 2009 stop-codon-eliminated variant). Forbidden internal restriction-site detection is an assembly-feasibility check used by Golden Gate / Gibson workflows.

The predicate registry merge in `validation_orchestrator.py` causes the `internal_sites.mr_22_forbidden_internal_sites` predicate to be invoked whenever rule MR-22 is evaluated. A user reading the audit trail will see "MR-22 fired" with a remediation hint about WPRE3, but the actual check that fired was about restriction-site forbiddenness — the two are not the same.

**Recommendation (v0.2.1):**
- Either rename the implemented predicate to a free predicate slot (e.g., `mr_synthesis_internal_sites`) and re-wire MR-39 / MR-43-style synthesis rules to it.
- Or land a real WPRE-detection predicate (sequence-similarity check between the construct and the canonical 633-nt wild-type WPRE / 247-nt WPRE3 mut6).
- And add a unit test that asserts the predicate's `__name__` and its parent rule's description are at least keyword-overlapping (terminator/WPRE/intron/marker/etc.).

### § 2.4 MR-59 marker–host auxotrophy check has a data gap — HIGH-severity

MR-59 (lines 2256–2300 in MR.yaml) says: "A yeast auxotrophic marker (URA3 / LEU2 / HIS3 / TRP1 / MET15 / LYS2) requires the corresponding host_genotype_requirement (e.g., ura3-Δ) in the host's genotype field. A marker→host mismatch is HARD because no selection pressure exists." Citation grade A1 (Sikorski & Hieter 1989, Brachmann 1998).

Validation passes through `MarkersCataloguePort.host_genotype_requirement`, and the markers catalogue at `catalogues/markers.yaml:247` correctly populates `host_genotype_requirement: "ura3-Δ (e.g., BY4741, BY4742, INVSc1)"` for URA3, `leu2-Δ0 / leu2-3,112` for LEU2, etc.

**But** the host catalogue `catalogues/hosts.yaml` records yeast hosts with placeholder genotype strings:

```yaml
- id: "host.scerevisiae_by4741"
  genotype: "S288C-derived haploid auxotrophic background; verify source genotype."
- id: "host.scerevisiae_by4742"
  genotype: "S288C-derived haploid auxotrophic background; verify source genotype."
- id: "host.scerevisiae_w303"
  genotype: "W303 lab background; verify source genotype."
```

The canonical BY4741 genotype is `MATa his3Δ1 leu2Δ0 met15Δ0 ura3Δ0` (Brachmann 1998). The canonical BY4742 is `MATα his3Δ1 leu2Δ0 lys2Δ0 ura3Δ0`. The canonical W303 is `MATa/α; leu2-3,112; his3-11; ade2-1; trp1-1; ura3; can1-100`. None of these tokens are in the host record's `genotype` field — only the placeholder string is.

**Impact:** MR-59 cannot machine-match. A simple substring search `"ura3-Δ" in host.genotype` returns False because the host string says only "S288C-derived...verify source genotype." A user adding a URA3-marked plasmid to a `BY4741` design will not be warned that the host carries `ura3Δ0` (which is in fact the correct host) — the rule simply can't fire because the data is missing.

**Recommendation (v0.2.1):** populate `genotype` with the canonical genotype string for every yeast strain. Tokens needed by MR-59 substring match: `ura3-Δ`, `leu2-Δ`, `his3-Δ`, `trp1-Δ`, `met15-Δ`, `lys2-Δ` (and equivalents `leu2-3,112`, `his3-11`, `his3-200`).

### § 2.7 MR-22 vs WPRE biology — what the rule should actually do

Since § 2.3 flags the predicate-name mismatch, this section documents what a correct WPRE detection predicate looks like, so v0.2.1 can implement it cleanly:

**Wild-type WPRE (Zufferey 1999)** is a 589-bp post-transcriptional regulatory element from woodchuck hepatitis virus. It contains an open reading frame for the X-protein (WHV X-Ag, an oncogene in hepatocellular carcinoma) that retains ~50 % of its expression in lentiviral vectors. The X-protein's transformation potential in murine and human cells has been demonstrated empirically (Themis 2005, Mol Ther).

**WPRE3 / WPRE-mut6 (Zanta-Boussif 2009)** is a 247-bp truncated variant that retains the gamma-1 + gamma-2 cis-elements responsible for the post-transcriptional enhancement effect, while eliminating the X-protein ORF via deletion of the X-protein start codon and most of its coding sequence. Functional expression enhancement in lentiviral vectors is preserved within 80–90 % of wild-type WPRE.

A correct MR-22 predicate is therefore a sequence-similarity check: does the construct contain a wild-type-WPRE-matching subregion (89–100 % identity over the X-protein ORF coding window 410–589 of wild-type WPRE)? If yes → recommend WPRE3 (mut6) replacement. The reference 247-bp WPRE3 sequence is available (the corpus `elements/introns_wpre/mut6.yaml` should carry it).

This implementation has nothing to do with restriction-site detection. The two `mr_22` predicate slots are biologically orthogonal.

### § 3.5 SV40 polyA record contains the BGH polyA sequence — CRITICAL mislabel

`docs/ml_corpus/records/elements/polyA/sv40.yaml` line 4 stores:
```
TTGTGCCTTCTAGTTGCCAGCCATCTGTTGTTTGCCCCTCCCCCGTGCCTTCCTTGACCCTGGAAGGTGCCACTCCCACTGTCCTTTCCTAATAAAATGAGGAAATTGCATCGCATTGTCTGAGTAGGTGTCATTCTATTCTGGGGGGTGGGGTGGGGCAGGACAGCAAGGGGGAGGATTGGGAAGACAATAGCAGGCATGCTGGGGA
```
(208 bp)

`docs/ml_corpus/records/elements/polyA/bgh.yaml` line 4 stores:
```
CTGTGCCTTCTAGTTGCCAGCCATCTGTTGTTTGCCCCTCCCCCGTGCCTTCCTTGACCCTGGAAGGTGCCACTCCCACTGTCCTTTCCTAATAAAATGAGGAAATTGCATCGCATTGTCTGAGTAGGTGTCATTCTATTCTGGGGGGTGGGGTGGGGCAGGACAGCAAGGGGGAGGATTGGGAAGACAATAGCAGGCATGCTGGGGATGCGGTGGGCTCTATGG
```
(225 bp)

The two sequences differ only in their first base (`T` vs `C`) and in the BGH record's additional 17-bp 3' extension (`TGCGGTGGGCTCTATGG`). They are otherwise byte-for-byte identical. **The SV40 polyA record is mislabeled; it stores the BGH polyA sequence with the trailing 17 bp truncated.**

The canonical SV40 late polyA signal is a different ~135-bp element from the SV40 genome around the late mRNA cleavage site (e.g., from pcDNA3 it is `AACTTGTTTATTGCAGCTTATAATGGTTACAAATAAAGCAATAGCATCACAAATTTCACAAATAAAGCATTTTTTTCACTGCATTCTAGTTGTGGTTTGTC...`). The Fiers 1978 SV40 paper (PMID 211141, cited in the record) does describe the actual SV40 genome including the polyA signal — but the stored sequence is NOT the SV40 polyA. It is the BGH polyA.

**Impact:** any user choosing the "SV40 polyA" record for a construct will silently get a BGH polyA cassette. Both are functional mammalian polyA signals, but they have different relative strengths (BGH > SV40 in most contexts) and different lengths (~225 bp BGH vs ~135 bp SV40 in canonical use). The construct will polyadenylate, but the user's metadata will lie about which polyA was used.

**Recommendation (v0.2.1):** fetch the canonical SV40 late polyA signal from a deposited record (e.g., pcDNA3 / pSV2-CAT) and replace the `bases` field in `sv40.yaml`. Or delete the duplicate record and add a `notes` field on `bgh.yaml` clarifying that SV40 polyA was the original target.

## § 4. Wet-lab protocol consistency findings

### § 4.1 The toolkit aligns with standard wet-lab cloning practice — PASS

The assembly engines emit `verification_checkpoints` and `expected_failure_modes` that match canonical bench protocols:

- **Gibson / NEBuilder HiFi:** 20-nt default overlap (canonical Gibson 2009 / NEB E2621), junction-Sanger verification, "insufficient_overlap" and "template_carryover" failure modes (both real — template carryover from PCR is the leading cause of background colonies).
- **In-Fusion:** 15-nt overlap (canonical Clontech / Takara product line), same verification + failure-mode pattern.
- **Golden Gate (BsaI-HFv2 / BsmBI-v2 / BbsI-HF):** 30 cycles 37°C/16°C, junction-Sanger + colony-PCR verification, "overhang_misligation" and "internal_type_iis_site" failure modes (both real — undomesticated internal Type IIS sites are the leading Golden Gate failure mode).
- **Restriction-ligation:** T4 DNA ligase, diagnostic-digest + Sanger verification, "vector_religation" and "wrong_insert_orientation" failure modes (both real — religation is the leading non-directional cloning background).
- **Gateway:** attB1/attB2 scar reporting (MR-37 frame compatibility + MR-38 scar-residue contribution), so the user knows the 8–10 aa N-scar and ~25 aa no-stop C-scar will appear in the translated product.
- **Yeast TAR:** present in the strategy enumeration; supports 80+-fragment assemblies for synthetic biology workflows.

### § 4.2 Strain-specific awareness is biologically sophisticated — STRONG PASS

MR-55..60 (the v0.2 Enrichment Amendment additions) demonstrate genuine strain-aware molecular biology:

- **MR-55 methylation-aware enzyme planning:** correctly tracks `dam`, `dcm`, `hsdRMS` host status; correctly lists Dam-blocked enzymes (XbaI, NheI, MboI, ClaI overlapping GATC), Dcm-blocked enzymes (EcoRII, MvaI, ApyI subset), and DpnI/DpnII template-state requirements. Suggested remediation includes switching to methylation-tolerant isoschizomers (XbaI → SpeI; EcoRII → PspGI) OR prepping the plasmid from a dam-/dcm- host (GM2163, JM110). This is canonical molecular biology straight out of REBASE and Sambrook 4th ed.
- **MR-56 disulfide-aware host:** ≥3 disulfide bonds + reducing cytoplasm → suggest Origami(DE3), Origami B(DE3), or SHuffle T7. References Bessette 1999 (PMID 10590025) and Lobstein 2012. Correct biology.
- **MR-57 protease-aware host:** PEST sequences / N-end-rule destabilising residues / IUPRED ≥30% disordered + lon+/ompT+ → suggest BL21 lineage. References Gottesman 1996 (PMID 8902478). Correct biology.
- **MR-58 rare-codon-aware host:** CAI < 0.5 OR windowed 6-codon rare-cluster scan ≥2 clusters + non-supplemented host → suggest Rosetta(DE3)/Rosetta(DE3)pLysS (pRARE-supplied tRNAs argU/argW/ileX/glyT/leuW/proL). References Kane 1995 (PMID 7493997). Correct biology.

### § 4.3 Pre-bench QC steps are catalogued — PASS

The assembly plans include `verification_checkpoints` tuples that map to bench QC workflows: junction-Sanger sequencing, colony PCR, diagnostic restriction digest. The MR rules collectively flag stop-codon strategy (MR-11 tandem stop), start-codon validity (MR-10), reading frame (MR-29 fusion junctions), polyadenylation placement (MR-15 premature, MR-52 mammalian Pol-II, MR-53 bacterial-Pol-II flag), assembly junction hairpins (MR-34), direct repeats (MR-43), homopolymer runs (MR-44), and GC content windows (MR-45, MR-46) — i.e., everything a Twist/IDT/GenScript synthesis quote tool would also flag.

A wet-lab user running through the v0.2 design path encounters these QC layers in the right order:

1. **Design-time (MR-01..60 + WR + SR + BR predicates):** validates the design against host biology before any DNA is ordered.
2. **Synthesis-time (MR-39 / MR-40 / MR-43 / MR-44 / MR-45 / MR-46):** flags features that would be rejected or marked "needs special handling" by gene-synthesis vendors. This saves real money — Twist returns synthesis quotes that flag GC-skewed regions, homopolymer runs, and direct repeats. The MR rules anticipate these flags.
3. **Assembly-time (assembly strategy `validate_parts` + `expected_failure_modes`):** flags assembly-chemistry-specific failure modes (Golden Gate internal Type IIS, Gibson insufficient overlap, restriction-ligation religation).
4. **Verification-time (`verification_checkpoints`):** lists the QC steps a wet-lab user should run after transformation — diagnostic digest, colony PCR, junction Sanger.

This is the canonical wet-lab cloning workflow, mapped through the toolkit. Every step has a corresponding catalogue entry or rule. **Strong PASS.**

### § 4.4 What is NOT catalogued — MEDIUM gap

The wet-lab consistency check found one nontrivial absence: **the corpus contains no entries for codon usage tables themselves.** `catalogues/hosts.yaml` carries a `codon_usage_table_id` field per host (e.g., `"ecoli_k12"`, `"s_cerevisiae"`) but the project does not ship the actual codon-usage table data. This is acceptable for v0.2 (MR-07 / MR-08 / MR-58 reference codon-tables as a configurable resource) but should be tracked as a v0.3 deliverable so MR-58's CAI<0.5 check has reference data to compute against.

Also absent from the modular-element corpus: secretion signal peptides (pelB, OmpA, DsbA for periplasmic bacterial; alpha-factor pre-pro for yeast; native human signal peptides for mammalian), degron tags (FKBP, AID, dTAG, ssrA), and riboswitches (theophylline, tetracycline aptamer). These are referenced indirectly via MR-28 and MR-48 but have no first-class corpus records.

5. **`length_bp` field equals byte length but doesn't equal CDS length** for partial records (§ 1.7). Documentation gap, not a code defect.

### § 6.4 Central-dogma compliance — PASS with one exception

Central dogma is honored throughout the catalogues and predicates. Restriction-site biology, replication-origin biology, transcription biology, translation biology, and post-translational biology all align with NCBI/Sambrook canonical biology.

The exception is the EYFP_AB860250 record (§ 1.5), which silently breaks the central-dogma "ATG..stop in frame" invariant for users in standard hosts because it carries a Mycoplasma-codon-table CDS without flagging it.

### § 6.5 Thermodynamics / ΔG / Tm — TRUSTED

The MR-12 RBS predicate accepts a `biology.rbs.leader_mfe_kcal_mol` metric and asserts `mfe > -10.0 kcal/mol` for accessibility. This matches Salis 2009 RBS Calculator's leader-region accessibility threshold (the negative ΔG indicates secondary structure; > -10 means accessible). MR-34 (junction hairpin) accepts a hairpin ΔG ≤ -9 kcal/mol threshold within 30 nt of a cloning junction — matches NEB / GenScript synthesis guidelines.

The MR-12 RBS rule's specific thresholds also include `5 ≤ spacing ≤ 13` nt between the Shine-Dalgarno motif (consensus AGGAGG) and the AUG. This matches the Shultzaberger & Schneider 2001 / Chen & Texada 2006 distributions of natural E. coli leader spacers (median = 8 nt, 95 % within 5–12 nt). The motif consensus `AGGAGG` is the strong-binding subset; the broader `AGGAGGT` / `GGAGGA` distribution would also be detected by a fuzzy-match implementation. The TIR ≥ 1000 threshold matches Reis & Salis 2020 RBS Calculator v2 strong-expression bucket.

These are thermodynamically realistic thresholds. The values are not the project's invention; they reflect published practice. **Strong PASS for thermodynamics.**

### § 6.6 Source verification — see § 3.4

PMIDs were spot-checked against PubMed. Four of six checked are wrong. See § 3.4 for the corrective sweep recommendation.

### § 6.7 Algorithmic verification — sequence-comparison, overhang optimisation

The Golden Gate `OverhangSetOptimiser` (used in `engine/assembly/golden_gate.py`) is described in `engine/overhang/` (not exhaustively read in this audit). The Pryor 2020 / Potapov 2018 overhang fidelity scoring is the canonical reference for Type IIS assembly design — it scores overhang sets by their pairwise mis-ligation propensity using NEB's empirical T4 ligase ligation-fidelity matrix. The thresholds `minimum_fidelity=0.90` and `minimum_per_overhang_fidelity=0.99` reflect the Pryor 2020 published recommendations for 15–35-fragment Golden Gate assemblies. **The algorithm choice is biologically appropriate.**

The compatibility-checker iteration over `iter_host_contexts(construct, hosts)` produces a `CompatibilityReport` with `host_results` per host context. The `passed` property returns True iff no issue's `blocked_gates` are non-empty. This is set-theoretic, deterministic, and easy to audit. The `to_payload` and `canonical_json` serializations enable byte-stable reproducibility across runs (important for ML training-data provenance). **Strong design.**

### § 6.8 Numerical-stability check — restriction-site coordinate math

`engine/sequence_analysis/find_sites.py` uses Python's integer arithmetic and `range(start, end)` half-open intervals throughout. No floating-point arithmetic. No off-by-one error possible in well-formed input. The `wraps_origin` flag is set when a match's start coordinate plus its length exceeds `len(body)` (where `body` is the doubled sequence for circular topology). The math is clean.

`engine/validation/predicates/cpg.py::mr_27_cpg_content_threshold` uses `(dna.count("CG") / max(1, len(dna))) * 100` — `max(1, ...)` defends against division by zero. The threshold `count_per_100bp > 8.0` is biologically a "CpG-rich region" (8 CG dinucleotides per 100 bp = 16 % CpG dinucleotide content, which is well above the human-genome baseline of ~1 CpG per 100 bp average outside islands). The Obs/Exp threshold `> 1.2` for `biology.cpg.observed_expected` corresponds to a strong methylation-susceptibility signal in mammalian promoters. **Both thresholds are biologically defensible.** Could be tightened to align with the Gardiner-Garden 1987 island definition (length > 200 bp, GC > 50 %, Obs/Exp > 0.6) but that is a v0.3 refinement, not a v0.2 defect.

## § 9. Open scientific questions for v0.3 cadence

1. **CAI reference distribution per host class.** What threshold of CAI is "biologically meaningful" vs "ML noise"? MR-07 uses [0.7, 0.95]; MR-58 uses < 0.5. Validate both thresholds against published expression-yield datasets (e.g., the Quax 2015 P. pastoris CAI/yield correlation, or the Boël 2016 E. coli CAI/yield map).

2. **Splice-site false-positive rate.** MR-16 uses SpliceAI ≥ 0.5 threshold. Above which threshold does SpliceAI produce more true positives than false positives on synthetic constructs? Worth validating against the Jaganathan 2019 SpliceAI publication's calibration data.

3. **AAV capacity hard limit.** MR-18 sets `AAV ≤ 4.7 kb between ITRs`. Should there be a soft warning at 4.5 kb? Empirical wet-lab data (e.g., Wu 2010 Mol Ther) suggests packaging efficiency drops sharply above 4.5 kb.

4. **WPRE3 vs wild-type WPRE risk-vs-benefit.** MR-22 says "if wild-type WPRE detected → recommend WPRE3." But WPRE3 (mut6, Zanta-Boussif 2009) trades expression strength for safety (lower X-protein retention). For non-clinical research vectors the wild-type may be preferable. Should the recommendation be context-aware?

5. **Inc-group classification for non-classical replicons.** MR-03 says "two plasmids in the same Inc group cannot co-exist stably." How does the toolkit classify newer engineered replicons (e.g., synthetic minimal pMB1-derivatives, R6K with engineered π-protein dependence)?

6. **Methylation-aware enzyme planning extension.** MR-55 handles Dam/Dcm/HsdRMS. Should it also handle EcoKI (CCWGG-specific) or N6-methyladenine systems? Likely v0.3 expansion.

7. **CpG island scoring methodology.** MR-27 uses `CpG count > 8/100 bp` as the warn threshold. Established CpG-island callers (e.g., Gardiner-Garden 1987, Takai & Jones 2002) use multi-criterion definitions (GC% > 50, Obs/Exp CpG > 0.6, length > 200 bp). Should MR-27 align with one of these published methodologies?

8. **Codon-pair bias vs single-codon CAI.** MR-07/MR-08 use CAI and %MinMax. Modern codon-optimisation also accounts for codon-pair bias (Coleman 2008 SAVE-style attenuation). Worth adding as MR-08b?

9. **mRNA secondary structure near translation start.** MR-12 covers RBS-region MFE for bacterial leaders. Mammalian uORF / 5' UTR structure also affects scanning. Worth adding an MR-12-mammalian variant?

10. **Position-specific bacterial codon usage near translation initiation.** First ~10 codons after ATG have a known different codon-usage profile in E. coli (favoring AT-rich codons to reduce mRNA secondary structure near RBS). Worth catalogue / rule support?
